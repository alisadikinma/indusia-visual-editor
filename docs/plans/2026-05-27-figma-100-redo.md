### Approach

**Figma MCP code-gen + manual polish, per-bundle commit.**

Per bundle:
1. `mcp__figma__get_design_context(nodeId, fileKey)` → component tree + tokens + reference code
2. `mcp__figma__get_screenshot(nodeId, fileKey)` → visual reference for manual diff
3. Translate to Vue 3 `<script setup lang="ts">` + Tailwind. Code-gen output is typically JSX-flavored — manual conversion mandatory.
4. Wire to existing Pinia stores + axios clients. **Zero backend changes.**
5. Update i18n EN+ID strings.
6. Update `data-testid` selectors. Re-target Vitest unit + Playwright e2e where needed.
7. Type-check (`vue-tsc --noEmit`), unit tests, e2e smoke. Green = commit + push to `main`.

## Risks

1. **Figma MCP code-gen quality** — output often JSX-flavored, attribute names may differ (`className` → `class`), event handlers may need conversion. Mitigation: manual review every code-gen output, never paste blindly.
2. **Test selector breakage** — current tests likely use Tailwind class names or text content. Mitigation: add `data-testid="..."` to every interactive element + structural region, re-target Vitest selectors to those.
3. **LSF chrome alignment** — LSF is third-party; we can only style around it. RegionDetailPanel + action strip must visually match Figma `199:2` without modifying LSF source.
4. **Bundle sequencing across context window** — 39 screens × code-gen call = expensive. Mitigation: clear context between bundles by relying on per-bundle plan doc references rather than chat memory.
5. **Engineer-mode toggle migration** — moving from PreferencesView to AppTopBar means current `useEngineerStore.toggle` callers need re-wiring. Mitigation: store API stays identical, only consumer location changes.
6. **i18n strings ballooning** — already 200+ keys; Figma copy might add another 100-200. Mitigation: organize by feature namespace (`gate1.*`, `gate2.*`, `eval.*`), no flat keys.
